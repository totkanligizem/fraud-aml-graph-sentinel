Modeling: fraud classification, investigation ranking, AML graph

Fraud detection model (classification)
Core objective: predict whether a 
transaction_event
 should be flagged as fraud.
Model baseline: - Gradient boosting (LightGBM / XGBoost‑style) is a strong baseline for tabular fraud; your
CV already uses LightGBM in ranking. (CV PDF, p. 2) - For imbalanced fraud, 
Precision‑Recall metrics
 are
central: the PR curve can be more informative than ROC in imbalanced scenarios. 
- Use 
Average Precision (AP / PR‑AUC)
; scikit-learn formally defines AP as a weighted mean of precision
values across thresholds. 
Imbalance strategies (implement at least two, compare): - Class weights / cost-sensitive loss (fast baseline). -
Downsampling negatives for training, but evaluate on untouched distribution. - Thresholding based on a
cost matrix (below), not on accuracy.
Calibration: - Implement Platt scaling or isotonic regression (as an optional module), because operational
risk thresholds care about calibrated probabilities.
Investigation queue prioritization (Learning‑to‑Rank)
This is where your portfolio becomes “bank-real”: an operations team does not only need a score; they need
a 
queue
.
You already have LambdaRank experience (CV PDF, p. 2). Use LightGBM’s ranking objectives: - LightGBM
supports ranking objectives such as 
lambdarank
 and 
rank_xendcg

rank_xendcg
 is faster with similar performance. 
- Ranking labels must be integers where higher = more relevant.

# LedgerSentinel — Fraud & AML Graph Intelligence Platform
LedgerSentinel is a production-style, portfolio-safe platform for:
1) Fraud detection (imbalanced classification),
2) AML typology + ring detection (graph analytics),
3) Investigation queue prioritization (Learning-to-Rank),
4) Monitoring + audit-ready explanations (drift + model cards),
5) A grounded Analyst Copilot (RAG with strict security controls).
> 
No real bank data is used. This repo ships only code + schemas + small 
synthetic demo data.
> 
Public/synthetic datasets are downloaded at runtime and mapped into a 
canonical `transaction_event` schema.
